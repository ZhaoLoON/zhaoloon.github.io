概述

Java1.4引入Java NIO(Non-blocking I/O 或者New I/O)，为Java提供了非阻塞IO，并且在Java 1.7中又引入了JAVA NIO.2，位Java增加了异步通道和新的文件心痛API。

  • Channels and Buffers

标准的IO是基于字节流和字符流进行操作，二NIO是基于Channel和Buffer进行操作，数据总是从通道读取到缓冲区中，或者从缓冲区写入到通道中。

  • Non-blocking IO

Java NIO让你以非阻塞的方式使用IO，例如：当线程从通道读取数据到缓冲区时，线程还是可以进行其他属性。当数据被写入到缓冲区时，线程可以继续处理它。从缓冲区写入通道也类似。

  • Selector

Java NIO引入了选择器的概念，选择器用于监听多个通道的事件(比如连接打开)。因此单个线程可以监听多个数据通道。需要使用Selector，需要向Selector注册Channel，然后调用它的select()方法。这个方法会一直阻塞到某个注册的通道有事件就绪。一旦这个方法返回，线程就可以处理这些事件。

  • Java NIO一些主要的Channel

    • FileChannel 从文件中读写数据
    • DatagramChannel 通过UDP读写网络数据
    • SocketChannel 通过TCP读写网络数据
    • ServerSocketChannel 可以监听新进来的TCP连接，对每个新进来的连接都会创建一个SocketChannel

  • Java NIO中关键的Buffer

    • ByteBuffer
    • CharBuffer
    • DoubleBuffer
    • FloatBuffer
    • IntBuffer
    • LongBuffer
    • ShortBuffer
    • MappedByteBuffer

Channel

Java NIO的Channel类似流，但又有些不同：

  • 既可以从通道中读取数据，又可以写数据到通道中。但流的读写通常是单向的。
  • 通道可以异步读写。
  • 通道中的数据总是要先读取一个Buffer，或者总是要从一个Buffer中写入。

Buffer

Buffer本质上是一块可以写入数据，然后可以从中读取数据的内存。这块内存被包装成NIO Buffer对象，并提供了一组方法，用于方便的访问块内存。

Buffer的基本用法

Buffer读写数据一下四个步骤：

  1. 写入数据到Buffer
  2. 调用flip()方法
  3. 从Buffer中读取数据
  4. 调用clear()方法或者compact()方法

向Buffer中写入数据时，buffer会记录下写了多少数据。一旦要读取数据，需要通过flip()方法将Buffer从写模式切换到读模式。在读模式下，可以读取之前写入的数据。
一旦读完所有数据，就需要清空缓存区，让它可以被再次写入。有两种方法清空缓冲区：

  • clear()方法会清空整个缓冲区。
  • compact()方法只会清空以读过的数据，任何未读的数据都被移到缓冲区的起始处，新写入的数据将放到缓冲区未读数据的后面。

Buffer的capacity, position和limit

理解Buffer的工作原理，需要熟悉它的三个属性：
下图转自并发编程网

  • capacity

    作为一个内存块，Buffer有一个固定的大小值，就是capacity。只能往里写capacity个byte、long、char等类型。一旦buffer满了，需要将其清空才能继续写数据。

  • position

    • 写模式：初始值为0，当没写入一个数据时，position向前移动到下一个可以插入的数据的Buffer单元。position最大值为capacity-1。
    • 读模式： 当Buffer从写模式切换到读模式，position会被重置为0，当从Buffer的position处读取数据时，position向前移动到下一个可以读的位置。

  • limit

    • 写模式：limit表示最多能往buffer里写多少数据，此时limit=capacity
    • 读模式：切换到读模式时，limit表示最多可以读到多少数据。此时limit会被设置成写模式下的position值。

Buffer的类型

Java NIO有以下Buffer类型

  • ByteBuffer
  • MappedByteBuffer
  • CharBuffer
  • DoubleBuffer
  • FloatBuffer
  • IntBuffer
  • LongBuffer
  • ShortBuffer

Buffer的分配

想要获得一个Buffer对象首先要进行分配。每个buffer有一个allocate方法。

向Buffer中写入数据

两种方式向Buffer中写入数据：

  • 从Channel写到Buffer
  • 通过Buffer的put()方法写到Buffer里。

flip()方法

flip方法将Buffer从写模式切换到读模式。调用flip方法会将position设回0，并将limit设置成之前position的值。
换句话说，position现在用于标记读的位置，limit表示之前写进了多少个byte、char等。

从Buffer中读取数据

两种方法从Buffer中读取数据：

  • 从Buffer读取数据到Channel
  • 使用get()方法从Buffer中读取数据

rewind()方法

rewind()方法将position设回0，可以重读Buffer中的数据。

clear()和compact()

一旦读完Buffer中的数据，需要让Buffer准备好再次被写入。
如果调用clear方法，position将被设回0，limit被设置成capacity的值。换句话说被清空了。Buffer中的数据并未清楚，只是这些标记告诉我们可以从哪里开始往Buffer里写数据。
如果Buffer中仍有未读的数据，且后续还需要这些数据，但是此时想要先写数据，那么使用compact方法。
compact方法将所有未读的数据拷贝到Buffer起始处。然后将position设到最后一个未读元素正后面。limit设置为capacity。

mark()与reset()方法

通过调用Buffer的mark方法，可以标记Buffer中的一个特定position。之后可以通过调用Buffer.reset()方法恢复到这个position。

equals与compareTo方法

equals()

满足一下条件，两个Buffer相等：

  1. 有相同的类型(byte、char、int等)
  2. Buffer中剩余的个数相等
  3. Buffer中所有剩余的byte、char等都相同

compareTo()

如果满足下列条件，则认为一个Buffer“小于”另一个Buffer：

  1. 第一个不相等的元素小于另一个Buffer中对应的元素 。
  2. 所有元素都相等，但第一个Buffer比另一个先耗尽(第一个Buffer的元素个数比另一个少)。

Scatter/Gather

  • Scatter 从Channel中读操作时将读取的数据写入多个buffer中。因此Channel将从Channel中读取的数据分散(scatter)到多个buffer中
  • Gather 写操作是将多个Buffer的数据写入同一个Channel，因此，Channel将多个Buffer中的数据聚集(gather)后发送到Channel

Scatter/Gather经常用于需要将传输的数据分开处理的场合，例如传输一个由消息头和消息体组成的消息，你可能会将消息体和消息头分散到不同的Buffer中，方便的处理消息头和消息体

Scattering Reads

Scattering Reads是指数据从一个channel读取到多个buffer中。

注意buffer首先被插入到数组，然后再将数组作为channel.read()的输入参数。read()方法按照buffer在数组中的顺序将从channel中读取的数据写入到buffer，当一个buffer被写满后，channel紧接着向另一个buffer中写。

Gathering Writes

Gathering Writes是指数据从多个buffer写入到同一个channel。

buffers数组是write()方法的入参，write()方法会按照buffer在数组中的顺序，将数据写入到channel，注意只有position和limit之间的数据才会被写入。因此与Scattering Reads相反，Gathering Writes能较好的处理动态消息。

通道之间的数据传输

两个FileChannel可以通过transferTo()和transferFrom()方法相互传输数据
transferFrom方法可以从一个channel传输数据到当前FileChannel中。

Selector

仅用单个线程来处理多个channel的好处是，只需要更少的线程来处理通道。对于操作系统来说，线程之间上下文切换开销很大，而且每个线程都要占用系统一些资源。

与Selector一起使用时，channel必须使用非阻塞模式。这意味着不能讲FileChannel与Selector一起使用，因为FileChannel不能切换到非阻塞模式，而套接字通道可以。

注意register方法的第二个参数。意思是Selector监听的事件，四种不同类型的事件：

  • Connect
  • Accept
  • Read
  • Write

通道触发了一个事件意思是该事件已经就绪。所以，某个channel成功连接到另一个服务器称为“连接就绪”。一个server socket channel准备好接收新进入的连接称为“接收就绪”。一个有数据可读的通道可以说是“读就绪”。等待写数据的通道可以说是“写就绪”。

四种事件用SelectionKey的四个常量表示：

  • SelectionKey.OP_CONNECT
  • SelectionKey.OP_ACCEPT
  • SelectionKey.OP_READ
  • SelectionKey.OP_WRITE

通过Selector选择通道

向Selector注册了通道后，可以调用select()方法，方法返回你所感兴趣的事件已经准备就绪的通道。

  • int select() 阻塞到至少一个通道在注册的事件上就绪
  • int select(long timeout) 增加超时时间
  • int selectNow() 不会阻塞，不管什么通道都立刻返回。如果从前一次选择操作后，没有一个通道就绪，则返回0

selectedKeys()

一旦select()方法返回，表明一个或多个通道已经就绪，然后可以通过selector.selectedKeys()方法，访问以就绪的事件。

wakeUp()

线程调用select()方法后进入阻塞状态，此时可以通过另一个线程调用Selector.wakeUp()方法，阻塞的select()方法会立即返回。
如果当前线程没有阻塞在select()方法，那么下个调用select()方法的线程会立即wakeUp。

close()

用完Selector后调用起close()方法会关闭该Selector，但通道本身并不会关闭。

SocketChannel

Java NIO中的SocketChannel是一个连接大TCP网络套接字的通道。

  • 打开一个SocketChannel并连接到网络上的某台服务器。
  • 一个新连接到达ServerSocketChannel时，会创建一个SocketChannel。

非阻塞模式

SocketChannel可以设置为非阻塞模式，非阻塞模式常常搭配Selector使用，想Selector注册一个或多个SocketChannel，设置之后可以异步调用connect(), read(), write()方法。

  • connect() 如果SocketChannel在非阻塞模式下，调用connect会在建立连接之前就返回，此时可以通过finishConnect()方法判断是否连接已经建立。
  • write() 非阻塞模式下write方法尚未写出任何数据时可能就返回了。需要在循环中调用write方法
  • read() 非阻塞模式下read方法在尚未读取到任何数据时可能就返回了。所以需要关注int返回值，它回告诉你读取了多少字节。

ServerSocketChannel

ServerSocketChannel可以监听新进来的TCP连接通道，就想表中IO中的ServerSocket。

非阻塞模式

ServerSocketChannel可以设置成非阻塞模式。在非阻塞模式下，accept()方法会立刻返回，如果还没有新进来来的连接，返回的将是null。

DatagramChannel

DatagramChannel是一个能首发UDP包的通道。因为UDP是无连接的网络协议，所以不能像其他通道那样被读取和写入，它发送和接受的是数据包。
